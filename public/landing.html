The Waiflo way

One JSON file.
The full pipeline.

Define steps, wire them together, run the whole thing with a single API call. Every workflow is a plain .waiflo.json file you can commit, review, and deploy like any other artifact.

How it works

Define once.
Run anywhere.

Three primitives. No magic, no hidden state, no framework to learn.

01
Steps — the atomic unit
Each step has a type (prompt, api, transform, tool, script), typed input/output schemas, and its full configuration. Reusable across pipelines.
ws_name · ws_type · ws_inputs_schema
02
Workflows — the DAG
Wire steps into a directed acyclic graph via wf_nodes and depends_on. The runtime resolves execution order. You define the intent.
wf_nodes · depends_on · step_id
03
Execute via REST API
One POST /api/exec/step call with inputs. SSE streaming for prompt steps. JSON response for API steps. Your backend stays in control — Waiflo is the engine, not the owner.
SSE streaming · JSON response · typed outputs

Design API

Surgical edits.
No clobbering.

PATCH endpoints use recursive deep merge — only the fields you send are updated. Built for autosave, inline editors, and AI-assisted pipeline builders that need to modify one field without touching the rest.

Current step
{
  "ws_name": "classify",
  "ws_llm": {
    "provider": "anthropic",
    "model": "claude-sonnet",
    "temperature": 0
  },
  "ws_system_prompt":
    "You are an expert…"
}
+
PATCH body
{
  "workflow_name": "triage",
  "ws_name": "classify",
  "patch": {
    "ws_llm": {
      "temperature": 0.4
    }
  }
}
→
Result
{
  "ws_name": "classify",
  "ws_llm": {
    "provider": "anthropic", ✓
    "model": "claude-sonnet", ✓
    "temperature": 0.4  ← patched
  },
  "ws_system_prompt":
    "You are…"  ✓
}
Step types

5 primitives.
Every use case covered.

Each step has a ws_type that determines runtime behavior, required fields, and execution semantics.

prompt
LLM call. System prompt, {{var}} template, typed output schema. Streams via SSE.
ws_llm
ws_system_prompt
ws_prompt_template
api
HTTP call. URL supports {{input}} substitution. Returns parsed JSON or raw text.
ws_api.method
ws_api.url
ws_api.headers

Get started

Up and running
in 3 minutes.

Self-host with Docker. Zero external services — just a JWT secret and your Anthropic key.

1
Clone and configure
Two secrets to generate: one for JWT signing, one for API key encryption at rest.
git clone https://github.com/jpdesmontils/waiflo
cd waiflo && cp .env.example .env
# Set JWT_SECRET and MASTER_SECRET in .env
2
Start the server
Spins up the API + visual editor. Persists workflow files to a mounted volume — no database required.
docker compose up -d
# Editor available at http://localhost:3001/editor
3
Open the editor — or call the API directly
Explore the visual editor without an account. Or register and start pushing workflows programmatically from your CI pipeline, your agent, or your backend.
# Try the editor first — no account needed
open http://localhost:3001/editor

# Or go API-first — register and push a workflow
curl -X POST http://localhost:3001/api/auth/register \
  -d '{"email":"you@example.com","password":"secret"}'
